# Документ 1: Глобальный Стейт, Навигация и Роутинг

## 1. Навигация и Роутинг

Переключение между экранами приложения сопровождается плавными анимированными CSS-переходами (fade-in / slide-эффекты):

1.  `/calendar` — Главный экран ежедневного новогоднего марафона (Табель-календарь).
2.  `/collider` — Экран настройки аналоговой панели и крафта новогодних украшений.
3.  `/inventory` — Экран физического хранилища игрока (сетка NxN) с механикой утилизации.
4.  `/collection` — Экран Книги коллекций (интерактивный архив-альбом со статистикой сборки по страницам).

---

## 2. Глобальная модель состояния (Энергонезависимый Стейт)

Вся бизнес-логика и данные сохраняются в **localStorage**. Состояние делится на две части: постоянную (сохраняется на диск) и временную (сбрасывается при обновлении страницы).

### Сущности данных (Domain Model):

_Примечание:_ Названия ключей и структура типизации являются рекомендуемыми (для единого нейминга во всем проекте), но финальная группировка и организация TS-интерфейсов остается за агентом.

- `userShards` — Текущий баланс осколков игрока (стартовое значение: 0).
- `chargesAvailable` — Количество бесплатных зарядов антиповторителя.
- `inventory` — Массив созданных предметов, находящихся у игрока. Обязательно должен содержать уникальный `id` предмета и время создания (`timestamp`).
- `calendar` — Состояние ежедневного табеля. Должно хранить текущий день прогресса (`currentDayIndex`) и время последнего получения награды (`lastClaimedTimestamp`).
- `unlockedCollectionIds` — Список идентификаторов уникальных игрушек, открытых в Книге коллекций.

_Примечание:_ Текущие положения роторов, тумблеров и выбранная кнопка в матрице являются **временным UI-состоянием** и в `localStorage` не записываются. При перезагрузке страницы прибор сбрасывается в дефолтный режим, но прогресс игрока остается неизменным. Запись на диск происходит строго в момент транзакций: забор награды, успешный крафт, разбитие вещи.
