## 🧱 Архитектура и UI

- **FSD:** соблюдение границ слоев и слайсов.
- **Публичный API:** межслайсовые импорты только через `index.ts`.
- **UI:** без сторонних UI-библиотек.

## 🔎 Типизация

- **Внешние данные:** сеть, `JSON.parse`, `localStorage`, `postMessage` и `catch` считаются `unknown` до проверки на границе приложения.
- **Сужение:** приоритет у проверок, type guards и discriminated unions. Guard проверяет всю форму, заявленную его предикатом.
- **Точность вывода:** `satisfies` используется для проверки конфигураций, `as const` — для сохранения литеральных типов.
- **Утверждения:** `any`, `as T` и `!` допустимы только локально при обоснованном инварианте или изоляции нетипизированного API. Они не должны скрывать неизвестную форму данных.
- **Подавление ошибок:** `// @ts-ignore` и `// @ts-nocheck` запрещены.

## 💾 Состояние и логика

- **Постоянное состояние:** прогресс игрока хранится в `localStorage`.
- **Временное состояние:** настройки панели и другое UI-состояние хранятся в React и сбрасываются при перезагрузке.
- **Владение логикой:** сценарии действий размещаются в `features`, инварианты сущностей — в `entities`, независимые переиспользуемые функции — в `shared/lib`.
- **Чистые функции:** вычисления отделяются от React и побочных эффектов. Чистота функции не определяет ее слой.

## 📝 Коммиты

- **Стандарт:** Conventional Commits.
- **Формат:** описание на русском языке, в нижнем регистре, с глаголом в прошедшем времени: `добавлен`, `исправлено`, `обновлена`.
- **Атомарность:** один коммит — одна логическая задача. Независимые изменения разделяются.
- **Scope:** необязателен. Используется, если у коммита есть один понятный смысловой владелец и scope делает сообщение точнее. При нескольких равнозначных областях scope не указывается.
- **`/commit`:** повторный анализ рабочей директории и предложение атомарных сообщений. Индексация и создание коммитов не выполняются.
- **Предложение:** после завершения задачи предлагается сообщение коммита.

## ✅ Проверки

- Проверки выполняются после завершения реализации.
- Изменения исходного кода: `pnpm type-check` и `pnpm lint`.
- Сборка, роутинг и интеграция модулей: дополнительно `pnpm build`.
- Документация: проверка diff, ссылок и непротиворечивости без запуска приложения.
