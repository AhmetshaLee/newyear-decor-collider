## 🧱 Архитектура и UI

- **Feature-Sliced Design (FSD 2.1+):** Строго соблюдай слои. Импортируй сущности из других слайсов только через публичные API (`index.ts`). Перекрестные импорты между фичами (cross-imports) запрещены.
- **Интерфейс:** Без UI-библиотек (пишем с нуля).

## 💾 Стейт и Логика

- **Типизация:** Запрещены `any` и директивы подавления ошибок (`// @ts-ignore`, `// @ts-nocheck`). Описывай сложные типы честно.
- **Разделение состояния:**
  - Постоянные данные (осколки, инвентарь, календарь, архив Книги коллекций) хранятся в `localStorage`.
  - Временное UI-состояние панели (углы роторов, выбранные кнопки, тумблеры, состояние антиповторителя) хранится только в оперативной памяти React и сбрасывается при перезагрузке.
- **Бизнес-логика:** Формулы расчета стоимости, фильтрацию пула наград и работу с датами выноси в чистые функции-хелперы в `shared/lib`.

## 📝 Правила коммитов (Conventional Commits)

После завершения задачи проанализируй итоговые изменения и предложи вариант сообщения коммита, даже если пользователь не вводил `/commit`. По запросу `/commit` повторно проанализируй текущее состояние рабочей директории:

- **Стиль:** Стандартный Conventional Commits.
- **Формат:** Текст описания пиши **на русском языке** в нижнем регистре. Используй глаголы в прошедшем времени («добавлен», «исправлено», «обновлена» и т. д.).
- **Атомарность:** Если изменения объемные или затрагивают разные независимые модули/фичи, обязательно разбей предложение на несколько последовательных атомарных коммитов (один коммит — одна логическая задача).
- **Scope:** Используй scope только когда он реально помогает понять область изменения. Сквозные, инфраструктурные, UI-общие и навигационные изменения можно оставлять без scope.

### Рекомендуемые scope

- `collider` — экран Коллайдера, холст, панель, элементы управления и zoom.
- `player` — сущность прогресса игрока и сохранение в `localStorage`.
- `craft-decor` — фича транзакции крафта украшения.
- `decors` — реестр украшений, значения альбомов/уровней/типов и фильтрация пула.
- `inventory` — нижняя панель и логика инвентаря.
- `collection` — правая панель книги коллекций и отображение прогресса.
- `calendar` — ежедневный календарь и логика наград.

Если изменение затрагивает несколько областей сразу, scope выбирается по основной области изменения: где больше смысловой нагрузки, риска или строк diff. Если основной scope неочевиден, агент должен явно перечислить варианты и коротко обсудить выбор с пользователем перед предложением `/commit`.

## ✅ Финальные проверки

- Запускай итоговые проверки после завершения реализации, а не после каждого промежуточного изменения.
- Для изменений исходного кода базовый набор — `pnpm type-check` и `pnpm lint`.
- `pnpm build` добавляется для изменений сборки, роутинга, интеграции модулей или по прямому запросу пользователя.
- Для документационных изменений не запускай проверки приложения без необходимости: проверь итоговый diff, ссылки и отсутствие противоречивых формулировок.
- Сначала заверши проверки и исправь найденные ошибки, затем предлагай сообщение коммита.
